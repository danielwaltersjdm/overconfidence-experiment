About This Study

Study 4 of an ongoing empirical research project on overconfidence in frontier large language models.

What we measure

Frontier AI models routinely produce confidence intervals alongside their predictions. A well-calibrated 90% confidence interval should contain the true outcome 90% of the time. If a model's stated 90% intervals only capture the actual value 60% of the time, the model is overconfident — it expresses more certainty than its accuracy warrants.

This study tracks that gap continuously and publicly, across four prediction domains and three time horizons.

Predictions / day

Each weekday at 16:30 UTC (after US market close), we query three AI models with the same set of 40 prediction items. Each model produces a point estimate and three confidence intervals (50%, 80%, 90%) for three future horizons (1-day, 1-week, 1-month).

A separate follow-up API call then collects the model's free-text reasoning for its prediction. The justification is always collected after the numeric prediction, never combined with it, to avoid the reasoning influencing the numbers.

When each prediction's horizon elapses, we fetch the resolved outcome from public data sources and score the prediction. The index updates daily.

The index metric

90% CI Hit Rate = proportion of predictions where the true outcome fell within the model's stated 90% confidence interval, over a rolling 30-day window.

Target: 90%. Below 90% = overconfident. Above 90% = underconfident.

Scoring formulas

Hit rate

For each confidence level c (50%, 80%, 90%), the hit rate is the proportion of predictions where:

CIc,low ≤ actual ≤ CIc,high

A perfectly calibrated model achieves hit rate = c at each level.

Expected Calibration Error (ECE)

ECE = mean( |stated coverage − empirical coverage| )

Averaged across the three CI levels (50%, 80%, 90%). Lower is better. ECE = 0 means perfect calibration.

Brier score (normalized)

Brier = ((point_estimate − actual) / reference_value)2

Measures point-estimate accuracy. Normalizing by a reference value (e.g., current stock price) makes the score comparable across items with different scales. Lower is better.

How predictions are collected

  1. Prediction call — the model receives the item name, current value, and date. It returns a structured JSON response with a point estimate and three confidence intervals (50%, 80%, 90%) for each of the three horizons.
  2. Justification call — a separate follow-up API call asks the model to explain its reasoning. This is always a distinct request, never combined with the prediction, to prevent the reasoning from influencing the numbers.

Each item is predicted once per model per day (no replications). All prompts include a frequency-framed calibration instruction asking the model to consider its interval widths in terms of how often outcomes should fall within them.

Ramp-up period

1-day outcomes are available starting Day 2 of collection. 1-week outcomes become available from Day 8. 1-month outcomes from Day 31. The index for each horizon is withheld until sufficient data has accumulated. During ramp-up, the dashboard shows a friendly countdown message instead of empty charts.

Limitations

  • Fixed item set — the same 40 items are predicted every day. Models may implicitly “learn” the item list over time, though each query is stateless (no conversation history).
  • Single daily snapshot — predictions are collected once per day, not continuously. Intraday volatility is not captured.
  • Model version pinning — API providers may silently update models behind fixed version strings. We log exact model IDs per prediction but cannot fully control for undisclosed upstream changes.
  • Free API rate limits — CoinGecko's free tier imposes rate limits. On rare occasions a crypto actual may not be retrieved; affected predictions are marked as failed and excluded from scoring.
  • No replication across API calls — each prediction is made once per item per model per day. Stochasticity in model outputs is not averaged out.

Data and code

All prediction files, actuals, scored results, and pipeline code are available in the project's GitHub repository. This site is a read-only view of those data files, updated automatically by a daily CI/CD pipeline.

This is Study 4 of an ongoing project. Studies 1–3 investigated overconfidence in 1-day equity predictions (Study 1), multi-horizon equity predictions (Study 2), and a multi-domain cross-section (Study 3). A research paper covering Studies 1–3 is in preparation.
